## [Unreleased]

### Security
- ES: `security_level` es ahora un campo configurable del `Profile` (`profiles.toml`) que se propaga a `nzpy.connect(securityLevel=...)`. Antes `connection.py` fijaba `securityLevel=1` (only-unsecured) para **todas** las conexiones: sesión y credenciales en claro, sin TLS. El **default pasa de `1` a `2`** (preferred-secured): negocia SSL y hace fallback a claro solo si el servidor no lo ofrece → seguro por defecto y sin romper on-prem sin TLS. Valores: `0` preferred-unsecured, `1` only-unsecured (opt-in explícito, solo lab), `2` preferred-secured (default), `3` only-secured (SSL requerido; recomendado para SaaS/nube). **Cambio observable**: perfiles que omiten el campo pasan a negociar SSL. Requiere validación humana antes de release. Witness E2E (`nzsaas`, `security_level=3`): conexión SSL OK (issue #136, ADR 0017).
- EN: `security_level` is now a configurable `Profile` field (`profiles.toml`) propagated to `nzpy.connect(securityLevel=...)`. Previously `connection.py` hardcoded `securityLevel=1` (only-unsecured) for **all** connections: session and credentials in cleartext, no TLS. The **default changes from `1` to `2`** (preferred-secured): negotiates SSL and only falls back to cleartext when the server offers no SSL → secure-by-default without breaking on-prem without TLS. Values: `0` preferred-unsecured, `1` only-unsecured (explicit opt-in, lab only), `2` preferred-secured (default), `3` only-secured (SSL required; recommended for SaaS/cloud). **Observable change**: profiles omitting the field now negotiate SSL. Requires human validation before release. E2E witness (`nzsaas`, `security_level=3`): SSL connection OK (issue #136, ADR 0017).

### Added
- ES: nueva tool `nz_switch_database` (modo `read`) que cambia la **base de datos de trabajo del perfil activo** reutilizando sus credenciales, para que las tools siguientes resuelvan nombres sin calificar contra ella (p. ej. moverse a `DESA_MAESTROBI` y consultar `DBO.EFE_MC_CREDITOS` sin prefijo de BD). Valida el identificador (`[A-Z][A-Z0-9_]*`) y que la BD sea visible en `_v_database`; si no, `OBJECT_NOT_FOUND` con la lista disponible. Actualiza el campo `database` del perfil activo en `profiles.toml` (persiste, como `nz_switch_profile`); el output da `previous_database` para volver. **No** cambia `host`/`user`/`mode`. Cambiar a la misma BD es no-op. Witness E2E (`nzsaas`): `DESA_MODELOS` → `DESA_MAESTROBI` → `SELECT COUNT(*) FROM DBO.EFE_MC_CREDITOS` = 39943 sin prefijo → restaurado. Sin nuevas dependencias (issue #153, ADR 0016).
- EN: new tool `nz_switch_database` (`read` mode) that switches the **working database of the active profile**, reusing its credentials, so subsequent tools resolve unqualified names against it (e.g. move to `DESA_MAESTROBI` and query `DBO.EFE_MC_CREDITOS` without a DB prefix). No new dependencies (issue #153, ADR 0016).
- ES: nueva tool `nz_call_procedure` (modo `admin`) que ejecuta un procedimiento vía `CALL schema.proc(args)` y devuelve el return code (`return_value`) más los mensajes `NOTICE`/`RAISE` del SP (`messages`, capturados de `cursor.notices`). Los argumentos van **parametrizados** (`?`, máx 100); `signature` opcional valida el nº de args del overload. `dry_run=true` por defecto devuelve el `call_sql` sin ejecutar; `confirm=true` con `dry_run=false` ejecuta. `timeout_s` opcional acota la conexión. `database` debe coincidir con el perfil activo y se aplica la guarda `assert_env_safe` (rechaza `CALL` a SPs `PROD_*` en perfiles no productivos). - EN: new tool `nz_call_procedure` (`admin` mode) that runs a procedure via `CALL schema.proc(args)` and returns the return code (`return_value`) plus the SP's `NOTICE`/`RAISE` messages (`messages`, captured from `cursor.notices`). La `signature` (lista de tipos de argumentos) es obligatoria porque Netezza desambigua overloads por ella; se valida token a token antes de interpolarse y el statement pasa por `sql_guard` (kind `DROP`). `if_exists=true` (default) hace no-op (`dropped=false`) cuando el procedimiento no existe, resuelto en la capa Python porque NPS no parsea `IF EXISTS` en `DROP PROCEDURE`. `database` debe coincidir con el perfil activo. - EN: new tool `nz_drop_procedure` (`admin` mode, `confirm` required) that drops a procedure overload via `DROP PROCEDURE schema.proc(types)`. The `signature` (argument-type list) is mandatory because Netezza disambiguates overloads by it; it is validated token by token before interpolation and the statement goes through `sql_guard` (kind `DROP`). `if_exists=true` (default) is a no-op (`dropped=false`) when the procedure does not exist, resolved in the Python layer because NPS does not parse `IF EXISTS` on `DROP PROCEDURE`. `database` must match the active profile. Applies to on-prem and the `nzsaas` cloud. No `dry_run` (mirrors `nz_drop_table`). No new dependencies (issue #149).

Nueva clave i18n `GUARD_REJECTED.PROD_REF_IN_NONPROD` (ES/EN). Sin nuevas dependencias (issue #147, ADR 0014).
- EN: new tool `nz_execute_ddl` (`admin` mode) that compiles a full caller-supplied `CREATE [OR REPLACE] PROCEDURE` (NZPLSQL) or `CREATE [OR REPLACE] VIEW` —inline via `sql` or read from `input_path`— against the active profile database (including the `nzsaas` cloud). `dry_run=true` (default) returns `sql_to_execute` without executing; `confirm=true` with `dry_run=false` compiles. `statement_type ∈ {procedure, view}` must match the DDL shape. Introduces the shared environment guard `sql_guard.assert_env_safe`: when the active profile database does not start with `PROD_`, any `PROD_*` identifier in the SQL is rejected with `GUARD_REJECTED` code `PROD_REF_IN_NONPROD` (prevents compiling production-targeting code from a development session; conservative fail-closed scan). `input_path` is read via `io/safe_read.py` (same path policy as `nz_export_ddl.output_path`, file ≤ 1 MiB). New i18n key `GUARD_REJECTED.PROD_REF_IN_NONPROD` (ES/EN). No new dependencies (issue #147, ADR 0014).
- ES: `nz_execute_ddl` gana el flag `allow_prod_reads: bool = False`. Con `true` **omite únicamente** la guarda `PROD_REF_IN_NONPROD`: el caller certifica que ya volteó todas las **escrituras** a la BD activa (no productiva) y que los `PROD_*` restantes son **solo lecturas** (habilita compilar contra `DESA_` un SP volteado que conserva `SELECT ... FROM PROD_x`). El resto de validaciones (statement único, cabecera, modo `admin`, `statement_type`, `sql_guard.validate`) se mantiene; aplica igual en `dry_run` y en compilación real. `assert_env_safe` no cambia (no hay rama nueva que reduzca estrictez en `sql_guard`); compilar un `CREATE` es inerte —las escrituras corren en `CALL`— y el default `false` conserva el bloqueo (falla cerrado). No se parsea read/write: el flag es la certificación explícita del caller (issue #156, ADR 0014 amendment 2026-07-09).
- EN: `nz_execute_ddl` gains the `allow_prod_reads: bool = False` flag. When `true` it **skips only** the `PROD_REF_IN_NONPROD` guard: the caller certifies it has already flipped every **write** target to the active (non-production) DB and that the remaining `PROD_*` refs are **read-only** (enables compiling a flipped SP against `DESA_` while it keeps `SELECT ... FROM PROD_x`). All other validations (single statement, header, `admin` mode, `statement_type`, `sql_guard.validate`) still apply; effective in both `dry_run` and real compilation. `assert_env_safe` is unchanged (no new `sql_guard` branch reduces strictness); compiling a `CREATE` is inert —writes run on `CALL`— and the default `false` preserves the block (fail-closed). Read/write is not parsed: the flag is the caller's explicit certification (issue #156, ADR 0014 amendment 2026-07-09).
- ES: `nz_export_ddl` admite dos parámetros opcionales nuevos asociados a `output_path` (issue #129): `include_resource_in_response: bool = False` controla si el bloque `EmbeddedResource` viaja en el response cuando se escribe a disco; `include_header: bool = True` controla si el archivo lleva un header `-- Database/Schema/Object/Exported …` seguido de `SET CATALOG <db>;`. `meta` gana tres campos: `preview` (primeras 10 líneas del archivo, sólo cuando el resource se omite), `resource_in_response` (bool, refleja si el response incluyó el resource) y `header_included` (bool, refleja si el archivo lleva el header). El header se construye en una función pura `build_header_block` (timestamp UTC ISO-8601 con sufijo `Z`, versión leída de `nz_mcp.__version__`, nombre del perfil como única identidad incluida; nunca host/user/password/connection string — anclado por test adversarial). Sin nuevas dependencias (`datetime` stdlib).
- EN: `nz_export_ddl` accepts two new optional parameters tied to `output_path` (issue #129): `include_resource_in_response: bool = False` controls whether the `EmbeddedResource` block ships in the response when writing to disk; `include_header: bool = True` controls whether the file is prefixed with a `-- Database/Schema/Object/Exported …` block followed by `SET CATALOG <db>;`. `meta` gains three fields: `preview` (first 10 lines of the file, only when the resource is omitted), `resource_in_response` (bool, reflects whether the response carried the resource), `header_included` (bool, reflects whether the file got the SET CATALOG header). The header is built by a pure `build_header_block` function (timestamp UTC ISO-8601 with `Z` suffix, version read from `nz_mcp.__version__`, profile name as the only identity-shaped field allowed; never host/user/password/connection string — anchored by an adversarial test). No new dependencies (stdlib `datetime`).
- ES: `nz_export_ddl` admite dos parámetros opcionales `output_path: str | None = None` y `overwrite: bool = False`. Cuando se pasa `output_path`, la tool persiste el DDL al filesystem del servidor MCP en `output_path` además de devolver el bloque resource y enriquece `meta` con `output_path`, `bytes_written` y `sha256` (hex SHA-256 del payload UTF-8). El archivo escrito es **byte-idéntico** al `text` del resource: UTF-8 sin BOM, sin reformateo, sin traducción CRLF (anclado por test). Política de paths validada antes de consultar Netezza: solo paths absolutos, sin segmentos `..`, sin `~`, sin caracteres de control ASCII; la carpeta padre debe existir (no se crea automáticamente); el archivo no debe existir salvo `overwrite=true`. En POSIX el archivo se crea con permisos `0600`; en Windows hereda la ACL del directorio padre (Python `Path.chmod` no setea bits POSIX en NT) — diferencia documentada en `docs/adr/0013-export-ddl-output-path.md`. Las violaciones de path / filesystem-state se devuelven con código estable `INVALID_INPUT` (detalle en `error.context.detail`). Sin `output_path` el comportamiento de la tool es idéntico al actual (back-compat estricta). Lógica de filesystem aislada en `src/nz_mcp/io/safe_write.py` con 100 % de cobertura unitaria. Witness E2E: `PROD_MAESTROBI.DBO.V_CONTINGENCIACREDITOSFULL` con perfil `uaipscrea1` (issue #127, ADR 0013).
- EN: `nz_export_ddl` accepts two new optional parameters `output_path: str | None = None` and `overwrite: bool = False`. When `output_path` is provided, the tool persists the DDL to the MCP server's filesystem at `output_path` in addition to returning the resource block, and enriches `meta` with `output_path`, `bytes_written` and `sha256` (hex SHA-256 of the UTF-8 payload). The file written is **byte-identical** to the resource `text`: UTF-8 without BOM, no reformatting, no CRLF translation (anchored by a regression test). Path policy validated before any catalog query: absolute paths only, no `..` segments, no `~`, no ASCII control characters; the parent directory must exist (auto-creation is intentionally disabled); the file must not exist unless `overwrite=true`. POSIX creates the file with `0600`; on Windows the ACL of the parent directory is inherited (Python's `Path.chmod` cannot set POSIX bits on NT) — documented in `docs/adr/0013-export-ddl-output-path.md`. Path / filesystem-state violations surface with the stable error code `INVALID_INPUT` (detail in `error.context.detail`). Without `output_path` the tool's behaviour is identical to the previous one (strict back-compat). Filesystem logic isolated in `src/nz_mcp/io/safe_write.py` with 100 % unit coverage. E2E witness: `PROD_MAESTROBI.DBO.V_CONTINGENCIACREDITOSFULL` with profile `uaipscrea1` (issue #127, ADR 0013).

### Fixed
- ES: `nz_insert` con 2+ filas y `on_conflict="error"` (el default) ya no falla contra Netezza. Antes `execute_insert` construía un `INSERT … VALUES (..),(..)` multi-fila que Netezza rechaza con `Multiple-row VALUES lists are not supported` (el `sql_guard` no lo detectaba porque parsea con dialecto postgres, que sí acepta esa sintaxis). Ahora emite una única sentencia atómica y parametrizada `INSERT INTO … (cols) SELECT ? … UNION ALL SELECT ? …`, la forma idiomática de inserción multi-fila en Netezza, verificada en vivo contra una instancia real (perfil `uaipscrea1`, `DESA_MODELOS`). La rama `on_conflict="skip"` no se toca. Un test unitario con un cursor que imita el rechazo de Netezza ancla la regresión (un `MagicMock` la ocultaba), más un smoke de integración `tests/integration/test_issue133_insert_multirow.py` (issue #133).
- EN: `nz_insert` with 2+ rows and `on_conflict="error"` (the default) no longer fails against Netezza. Previously `execute_insert` built a multi-row `INSERT … VALUES (..),(..)` that Netezza rejects with `Multiple-row VALUES lists are not supported` (the `sql_guard` missed it because it parses with the postgres dialect, which accepts that syntax). It now emits a single atomic, parameterized statement `INSERT INTO … (cols) SELECT ? … UNION ALL SELECT ? …`, the idiomatic Netezza multi-row insert, verified live against a real instance (profile `uaipscrea1`, `DESA_MODELOS`). The `on_conflict="skip"` branch is untouched. A unit test with a cursor that mimics Netezza's rejection pins the regression (a `MagicMock` hid it), plus an integration smoke `tests/integration/test_issue133_insert_multirow.py` (issue #133).
- ES: `nz_export_ddl(..., output_path=…)` ya no choca con el cap del MCP server (~100 KB / 25k tokens) cuando el DDL es grande. Reproducción real (2026-05-08, perfil `uaipscrea1`): `nz_export_ddl(procedure, PROD_ANALITICA, DBO, PI_CLIENTESTCM, output_path=…)` con un DDL de ~67 KB devolvía `Error: result … exceeds maximum allowed tokens` aunque el archivo se hubiera escrito correctamente. Causa: el response incluía el `EmbeddedResource` con todo el `text` del DDL más el summary y la metadata, lo que sumado al overhead JSON superaba el cap. Ahora el response **omite** el bloque resource por default cuando hay `output_path` (el archivo es la fuente de verdad) y reporta un `preview` con las 10 primeras líneas del archivo. El parámetro `include_resource_in_response: bool = False` permite restablecer la forma anterior bajo riesgo conocido. Adicionalmente, los archivos escritos llevan ahora un header `-- Database/Schema/Object/Exported …` + `SET CATALOG <db>;` (parametrizable con `include_header`) para que sean auto-contenidos y re-ejecutables (issue #129, ADR 0013 revisión 2026-05-08).
- EN: `nz_export_ddl(..., output_path=…)` no longer collides with the MCP server cap (~100 KB / 25k tokens) for large DDLs. Real reproduction (2026-05-08, profile `uaipscrea1`): `nz_export_ddl(procedure, PROD_ANALITICA, DBO, PI_CLIENTESTCM, output_path=…)` with a ~67 KB DDL returned `Error: result … exceeds maximum allowed tokens` even though the file was successfully written. Cause: the response carried the full `EmbeddedResource` text plus summary and metadata, exceeding the cap once JSON overhead was added. The response now **omits** the resource block by default when `output_path` is set (the file is the source of truth) and reports a `preview` with the first 10 lines of the file. The new `include_resource_in_response: bool = False` parameter lets callers restore the previous shape at their own truncation risk. Additionally, files now ship with a `-- Database/Schema/Object/Exported …` + `SET CATALOG <db>;` header (toggle via `include_header`) so they are self-contained and re-executable (issue #129, ADR 0013 revision 2026-05-08).

### Changed
- ES: `nz_export_ddl` cambia el **default** de la respuesta cuando se pasa `output_path`: el bloque `EmbeddedResource` ya no viaja por default; sólo summary + meta + `preview`. Para callers de #128 que dependían de leer `content[0].resource.text` con `output_path` seteado, pasar `include_resource_in_response=True` recupera la forma previa (asumiendo el riesgo de cap collision). El sha256 reportado es ahora el del archivo en disco (incluye el header cuando `include_header=true`); los callers que necesiten byte-identidad estricta entre archivo y resource deben pasar `include_header=False` (issue #129, ADR 0013 revisión).
- EN: `nz_export_ddl` changes the **default** response shape when `output_path` is supplied: the `EmbeddedResource` block no longer ships by default; only summary + meta + `preview`. Callers from #128 that relied on reading `content[0].resource.text` with `output_path` set should pass `include_resource_in_response=True` to restore the previous shape (accepting the cap-collision risk). The reported sha256 is now the digest of the file on disk (covers the header when `include_header=true`); callers needing strict byte-identity between file and resource should pass `include_header=False` (issue #129, ADR 0013 revision).

### Fixed
- ES: `nz_get_view_ddl` y `nz_export_ddl(object_type='view')` ahora devuelven el DDL real cuando la vista vive en una BD distinta a la del perfil activo. Antes Netezza proyectaba el sentinel literal `'Not a view'` en `_V_VIEW.DEFINITION` para cualquier consulta cross-database, porque la columna se calcula lazy desde `_T_RULE` del catálogo *actual* de la sesión y el join silencioso fallaba al cruzar bases. El fix emite `SET CATALOG <database>` en la conexión efímera de la tool justo antes del `SELECT` contra `<BD>.._V_VIEW`; cada llamada abre y cierra su propia conexión nzpy, así que el cambio de catálogo no contamina queries posteriores. El identificador de BD se valida con `validate_database_identifier` (alfabeto `[A-Z][A-Z0-9_]*`) antes de interpolarse en el `SET CATALOG` (Netezza no admite parámetros bind en esta sentencia). `nz_get_table_ddl` y `nz_get_procedure_ddl` no se tocan: ya funcionan bien cross-DB porque reconstruyen desde catálogos seguros (`_v_relation_column`, `_v_procedure`). Witness E2E: `PROD_MAESTROBI.DBO.V_CONTINGENCIACREDITOSFULL` con perfil `uaipscrea1` (default DB `DESA_MODELOS`) (issue #125).
- EN: `nz_get_view_ddl` and `nz_export_ddl(object_type='view')` now return the real DDL when the view lives in a database different from the active profile. Previously Netezza projected the literal sentinel `'Not a view'` into `_V_VIEW.DEFINITION` for any cross-database lookup, because the column is computed lazily from `_T_RULE` of the session's *current* catalog and the silent join failed across databases. The fix emits `SET CATALOG <database>` on the tool's ephemeral connection right before the `SELECT` against `<BD>.._V_VIEW`; each call opens and closes its own nzpy connection, so the catalog change does not leak into subsequent queries. The DB identifier is validated by `validate_database_identifier` (alphabet `[A-Z][A-Z0-9_]*`) before being interpolated into the `SET CATALOG` (Netezza does not accept bind parameters there). `nz_get_table_ddl` and `nz_get_procedure_ddl` are untouched: they already work cross-DB by reconstructing from safe catalogs (`_v_relation_column`, `_v_procedure`). E2E witness: `PROD_MAESTROBI.DBO.V_CONTINGENCIACREDITOSFULL` with profile `uaipscrea1` (default DB `DESA_MODELOS`) (issue #125).
- ES: `nz_list_tables`, `nz_list_views`, `nz_list_procedures`, `nz_list_schemas`, `nz_list_databases`, `nz_get_procedures_ddl_batch` y `nz_find_table_references` ahora hacen match case-insensitive sobre el parámetro `pattern`. Antes el `LIKE` se evaluaba contra el case original del input, y como Netezza normaliza los nombres del catálogo a mayúsculas, un patrón con minúsculas devolvía 0 filas aunque el objeto existiera (`pattern='EFE_MC_codigogestion'` → `[]`, `pattern='EFE_MC_CODIGOGESTION'` → 1 fila). Las queries en `catalog/queries.py` ahora envuelven el placeholder con `UPPER(?)`, manteniendo la semántica de wildcards (`%`/`_`) que el usuario haya escrito (issue #123).
- EN: `nz_list_tables`, `nz_list_views`, `nz_list_procedures`, `nz_list_schemas`, `nz_list_databases`, `nz_get_procedures_ddl_batch` and `nz_find_table_references` now match the `pattern` argument case-insensitively. Previously the `LIKE` ran against the original-case input, but Netezza stores catalog names in upper case, so a lower-case pattern returned 0 rows even when the object existed (`pattern='EFE_MC_codigogestion'` → `[]`, `pattern='EFE_MC_CODIGOGESTION'` → 1 row). The catalog SQL in `catalog/queries.py` now wraps the placeholder with `UPPER(?)`, preserving the wildcard semantics (`%`/`_`) the user supplied (issue #123).
- ES: `parse_sections` ahora detecta secciones en SPs cuyos literales contienen caracteres de salto de línea (CR, VT, FF, NEL, U+2028, U+2029) o cuyos comentarios `--` / `/* */` contienen apóstrofos sueltos. Antes `mask_single_quoted_strings` colapsaba esos caracteres dentro de literales y `splitlines()` del masked source quedaba más corto que el del source crudo, lo que hacía que la búsqueda del `END;` exterior perdiera líneas (`sections_detected: []` en SPs con CR). Adicionalmente, un `'` dentro de un comentario abría una "literal fantasma" que blanqueaba el resto del cuerpo. Nuevo helper `mask_literals_preserving_lines` enmascara contenido de literales y de comentarios pero preserva todos los caracteres de boundary, garantizando `len(masked.splitlines()) == len(source.splitlines())`.
